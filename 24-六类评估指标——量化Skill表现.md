> 学习目标：掌握六类核心评估指标，量化Skill表现

---

## 引言

上一节我们学会了两类评分器：确定性检查抓底线，评分细则检查抓质量。

但有了评分还不够，**你还需要量化指标来持续监控 Skill 的表现**。

你是不是也遇到过这种情况：

- 改了 Skill，通过了测试，但不知道 Token 消耗是不是变高了
- Skill 一直在用，但不知道执行速度是不是变慢了
- 某个 Skill 重试次数突然变多，但不知道是哪次改坏的

**根本原因**：只关注"能不能过"，忽略了"稳不稳、贵不贵、快不快"。

**解决**：用六类评估指标量化 Skill 的真实表现。

---

## 📊 评分器与评估指标的关系

**评分器是"评估工具"，评估指标是"量化维度"**。

### 两者的分工

| 评分器 | 评估指标 |
|--------|----------|
| **作用**：判断单个测试用例是否通过 | **作用**：统计整体测试结果 |
| **类型**：确定性检查 + 评分细则检查 | **类型**：六类量化维度 |
| **输出**：每个测试用例的通过/不通过 | **输出**：整体表现的数值统计 |
| **回答的问题**：这个测试用例做得对不对？ | **回答的问题**：Skill整体表现怎么样？ |

### 评估流程

```
测试用例 → 评分器判断 → 通过/不通过 → 统计六类指标 → 得出整体得分
   ↓           ↓              ↓              ↓              ↓
 50个测试   确定性检查      35个通过      通过率70%      等级B
           评分细则检查    15个失败      Token消耗
                                      执行时长...
```

### 具体示例

**场景**：评估一个PDF Skill

**评分器的工作**：
- 测试用例1：填充PDF表单 → 确定性检查：是否创建了填充后的PDF？ → ✅ 通过
- 测试用例1：评分细则检查：字段填写是否准确？ → ✅ 通过
- 测试用例2：填充PDF表单 → 确定性检查：是否创建了PDF？ → ❌ 失败
- ...（50个测试用例）

**评估指标的工作**：
- 通过率：35/50 = 70%
- Token消耗：平均12500 tokens/测试
- 执行时长：平均45秒/测试
- 重试率：15次重试/50测试 = 30%
- ...

### 为什么需要评估指标？

**评分器只能告诉你"能不能过"，评估指标能告诉你"稳不稳、贵不贵、快不快"**。

| 只有评分器 | 有评分器 + 评估指标 |
|-----------|-------------------|
| 知道35/50通过 | 知道通过率70%、成本、速度 |
| 不知道Token消耗 | Token消耗一目了然 |
| 不知道执行速度 | 执行时长、命令次数一目了然 |
| 不知道稳定性 | 重试率、错误分布一目了然 |

**核心关系**：评分器是评估的基础，评估指标是评估的升华。

---

## 📚 核心问题

**怎么量化 Skill 的表现，不只是看通过率。**

常见问题：

- 只盯着通过率，忽略了成本和速度
- Token 消耗变高，才发现 Skill 写得太复杂
- 执行时长变慢，才发现命令执行次数太多

---

## 🔧 六类评估指标

### 1. **通过率**（Pass Rate）

测试用例中有多少通过了。

**如何计算**：
```python
pass_rate = passed_tests / total_tests * 100
```

**示例**：
- Skill v1: 35/50 通过 → 70% 通过率
- Skill v2: 42/50 通过 → 84% 通过率
- 提升：+14%

**价值**：最基础的指标，判断 Skill 是否能用。

---

### 2. **Token 消耗**（Token Cost）

平均每个测试用例消耗了多少 token。

**如何计算**：
```python
avg_tokens = total_tokens / total_tests
```

**示例**：
- Skill v1: 平均 12500 tokens/测试
- Skill v2: 平均 8900 tokens/测试
- 节省：-29%

**价值**：成本指标。Token 消耗高，说明 Skill 写得太复杂，或者让 Agent 做了太多不必要的事。

---

### 3. **执行时长**（Execution Time）

平均每个测试用例执行了多久。

**如何计算**：
```python
avg_time = total_time / total_tests
```

**示例**：
- Skill v1: 平均 45 秒/测试
- Skill v2: 平均 32 秒/测试
- 提速：-29%

**价值**：效率指标。执行时长过长，影响用户体验，或者说明命令执行次数太多。

---

### 4. **命令执行次数**（Command Executions）

每个测试用例平均执行了多少次 bash 命令。

**如何计算**：
```python
avg_commands = total_commands / total_tests
```

**示例**：
- Skill v1: 平均 12 次命令/测试
- Skill v2: 平均 8 次命令/测试
- 减少：-33%（避免重复命令）

**价值**：效率指标。命令次数多，说明 Skill 步骤太多，或者有重复执行。

---

### 5. **重试次数**（Retries）

模型因为失败重试了多少次。

**如何计算**：
```python
retry_rate = retries / total_tests
```

**示例**：
- Skill v1: 15 次重试 / 50 测试 → 30% 重试率
- Skill v2: 3 次重试 / 50 测试 → 6% 重试率
- 改善：-80%

**价值**：稳定性指标。重试次数多，说明 Skill 容易失败，不够稳定。

---

### 6. **错误类型分布**（Error Distribution）

不同类型错误的出现频率。

**如何计算**：
```python
error_distribution = {
    "type_error": count,
    "timeout_error": count,
    "assertion_error": count,
    ...
}
```

**示例**：
- Skill v1: 类型错误 12 次，超时 8 次，断言失败 5 次
- Skill v2: 类型错误 3 次，超时 1 次，断言失败 2 次
- 改善：所有错误类型都减少

**价值**：诊断指标。看错误类型分布，能知道 Skill 在哪里容易出问题。

---

## 💡 为什么需要六类指标？

**只看通过率不够**：

| 只看通过率 | 看六类指标 |
|----------|----------|
| 知道能不能过 | 知道稳不稳、贵不贵、快不快 |
| 发现不了成本上升 | Token 消耗一目了然 |
| 发现不了速度变慢 | 执行时长、命令次数一目了然 |
| 发现不了不稳定 | 重试率、错误分布一目了然 |

**核心思想**：评估不只是看结果，还要看成本、效率、稳定性。

---

## 📊 指标对比示例

**案例：优化 API 文档生成 Skill**

| 指标 | Skill v1 | Skill v2 | 变化 | 诊断 |
|------|----------|----------|------|------|
| 通过率 | 70% | 84% | ✅ +14% | 整体变好 |
| Token 消耗 | 12500 | 8900 | ✅ -29% | 成本下降 |
| 执行时长 | 45s | 32s | ✅ -29% | 速度提升 |
| 命令次数 | 12 | 8 | ✅ -33% | 步骤精简 |
| 重试率 | 30% | 6% | ✅ -80% | 稳定性提升 |

**结论**：Skill v2 不只通过了更多测试，还更省、更快、更稳。

---

## 🎯 本节核心观点

**六类指标的三个层次**：

1. **结果层**：通过率 → Skill 是否能用
2. **成本层**：Token 消耗 → Skill 是否省钱
3. **效率层**：执行时长、命令次数、重试率 → Skill 是否快、稳

**核心原则**：不只看结果，还要看成本、效率、稳定性。
