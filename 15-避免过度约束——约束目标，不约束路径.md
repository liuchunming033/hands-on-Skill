
> 学习目标：理解过度约束的危害，给Agent足够的自由度

---


## 引言

你是不是也遇到过这种情况：Skill 调用了，但输出效果很差？

你检查了内容，发现每一步都写得很详细，但 Agent 就是做不好。

问题可能出在：**步骤写得太死，失去了灵活性。**

## 核心内容

### 问题诊断：为什么 Skill 调用后效果差？

**常见原因**：过度约束，把步骤写死了。

**错误写法**：

```markdown
## 代码部署流程

1. 先运行 npm test
2. 再运行 npm run lint
3. 最后运行 npm run build
4. 部署到生产环境
```

看起来很详细，但问题来了：

- 如果测试失败，要继续吗？
- 如果 lint 有警告，要修复吗？
- 如果构建时间很长，要等吗？
- 顺序一定要这样吗？

Agent 变成了一个只会执行死命令的机器人，不会思考。

### 正确做法：约束目标

**正确写法**：

```markdown
## 代码部署流程

在部署前，确保代码通过测试、规范检查、构建。根据项目情况调整执行顺序和策略。

关键检查点：
- 所有测试通过
- 无 lint 错误
- 构建成功

如果发现问题，先修复再部署。
```

**核心原则**：**约束目标，而不是约束路径。**

### 约束目标 vs 约束路径

| 方式 | 特点 | 结果 |
|------|------|------|
| 约束路径 | 写死每一步 | Agent 变机器人，不会变通 |
| 约束目标 | 定义最终结果 | Agent 会思考，灵活应对 |

### Anthropic 案例：代码审查

**错误写法**：

```markdown
## 代码审查

1. 检查命名规范
2. 检查代码风格
3. 检查注释完整性
4. 检查测试覆盖率
```

**问题**：不同项目有不同的重点，步骤写死就不灵活。

**正确写法**：

```markdown
## 代码审查

审查代码质量，关注安全、性能、可维护性。根据项目类型调整审查重点：

- 前端项目：关注性能、可访问性
- 后端项目：关注安全、数据验证
- 库项目：关注 API 设计、文档完整性

输出结构化审查结果，标注严重程度。
```

### 给 Agent 灵活性

**不要**：

- 写死每一步的顺序
- 固定每个操作的参数
- 规定必须用哪个工具

**要**：

- 定义最终要达到的目标
- 给出关键检查点
- 让 Agent 自己决定怎么做

### 如何判断是否过度约束？

问自己三个问题：

1. **这个步骤顺序是必须的吗？**
2. **这个参数是固定的，还是可以灵活调整？**
3. **不同情况下，做法可能不同吗？**

如果答案是"可以灵活"，就不要写死。

## 总结

今天我们学习了"避免过度约束"：

1. **过度约束会让 Agent 变成机器人**，不会思考变通
2. **约束目标，不约束路径**：定义最终结果，让 Agent 自己决定怎么做
3. **给 Agent 灵活性**：根据不同情况调整策略

**核心原则**：给 Agent 它需要的信息，但也要给它适应不同情况的灵活性。
