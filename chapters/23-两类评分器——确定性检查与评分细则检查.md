
> 学习目标：学会确定性检查和评分细则检查

---


## 引言

上一节我们立起了七步生命周期：从决定到扩大，把写 Skill 变成可迭代流程。

这一节要解决一个更实操的问题：**评分器怎么构建？**

更新了 Skill，跑完看起来没问题——但怎么知道真的变好了，没把别处搞坏？

OpenAI 给了一个清晰的答案：**两种评分器分工，确定性检查抓底线，评分细则检查抓质量**。

关键：**你收集的是评估数据，不是审稿意见**。

## 核心内容

### 三步构建评分器

构建评分器不是一步到位，而是分三层：

```
第一步：记录运行轨迹
         ↓
第二步：确定性检查（抓底线）
         ↓
第三步：评分细则检查（抓质量）
```

### 第一步：记录运行轨迹

**工具**：用 `codex --trace-json` 参数，Skill 每执行一步输出一条结构化记录。

**记录内容**：

- 跑了什么命令
- 创建了什么文件
- 全部按时间顺序记下来

**为什么需要运行轨迹**：

| 没有运行轨迹 | 有了运行轨迹 |
|-------------|-------------|
| 只能看最终产物 | 能看到每一步发生了什么 |
| 不知道哪里出问题 | 能精确定位出问题的步骤 |
| 只能人工复盘 | 可以程序化分析 |

**价值**：不是日志摘要，是完整的可被程序解析的运行轨迹。有了运行轨迹，评分器才有东西可以解析。

### 第二步：确定性检查（Deterministic Check）

**定义**：纯代码规则，不用大模型，写死判断逻辑。

**回答的问题**：基础动作做了没有？

**示例**：

- 有没有跑 `npm install`？
- 有没有创建 `package.json`？
- 命令顺序是否符合预期？

**两个关键特性**：

| 特性 | 说明 |
|------|------|
| **确定性** | 同样行为每次判断结果一样，没有模糊地带 |
| **可调试** | 失败后打开记录文件能看到从哪一步开始偏 |

**价值**：给 Skill 装了飞行记录仪。

**上限**：能告诉动作发生了没有，告诉不了做出来的东西是不是你想要的样子。

**确定性检查示例**：

```python
# 检查是否创建了 package.json
def check_package_json_created(trace):
    for step in trace:
        if step["action"] == "create_file" and step["path"] == "package.json":
            return True
    return False

# 检查是否运行了 npm install
def check_npm_install_run(trace):
    for step in trace:
        if step["action"] == "run_command" and "npm install" in step["command"]:
            return True
    return False
```

### 第三步：评分细则检查（Rubric Check）

**定义**：用 LLM 做裁判，让它读仓库打分。

**适用场景**：验收标准本来就不是硬规则。

- 代码结构够不够干净？
- 组件组织是否符合团队约定？
- 样式配置是否按既定风格落地？

**评分细则（Rubric）是什么**：一张拆解表，把模糊质量标准拆成若干具体可独立判断的维度。

**例子**："代码质量好"拆成：

| 维度 | 判断标准 | 分值 |
|------|---------|------|
| 目录结构 | 组件是否按功能分目录 | 0-10 |
| 未使用引用 | 是否有未使用的 import | 0-10 |
| 命名约定 | 样式类名是否遵循命名约定 | 0-10 |
| 配置管理 | 配置项是否集中管理 | 0-10 |

每个维度单独打分，最后汇总。

---

### 具体实施步骤

针对代码风格、目录结构、配置规范等偏主观的要求，采用这套方案：

**三步流程**：

```
第一步：运行 Skill 生成产物
         ↓
第二步：让智能体按照预设评分细则检查成果
         ↓
第三步：限定输出为固定 JSON 格式，包含总分、分项结果、问题备注
```

**命令示例**：

检查仓库并输出符合评分标准的 JSON 响应：

```bash
# 运行评分细则检查
codex eval \
  --skill my-feature-skill \
  --rubric-file rubrics/code-quality.json \
  --output-format json \
  --output-file evaluation_result.json

# 查看评分结果
cat evaluation_result.json
```

**rubrics/code-quality.json 示例**：

```json
{
  "rubric_name": "代码质量评分细则",
  "dimensions": [
    {
      "name": "目录结构",
      "description": "组件是否按功能分目录",
      "max_score": 10,
      "check_command": "检查 src/ 目录结构"
    },
    {
      "name": "未使用引用",
      "description": "是否有未使用的 import",
      "max_score": 10,
      "check_command": "检查代码中的 import 语句"
    },
    {
      "name": "命名约定",
      "description": "样式类名是否遵循命名约定",
      "max_score": 10,
      "check_command": "检查 CSS 类名和变量命名"
    },
    {
      "name": "配置管理",
      "description": "配置项是否集中管理",
      "max_score": 10,
      "check_command": "检查配置文件组织"
    }
  ],
  "passing_threshold": 7.0
}
```

**执行过程**：

1. **运行 Skill**：执行 Skill 生成代码仓库
2. **智能体检查**：根据 rubric 检查每个维度
3. **输出 JSON**：固定格式，便于统计和对比

---

### 关键：输出格式必须被限定

**为什么要严格限定格式？**

提前写好模板，规定大模型必须输出哪些字段：

- 整体是否通过
- 总分
- 每个维度得分
- 为什么这么打分

大模型必须按模板填空，不能自由发挥。

**输出格式 Schema 示例**：

```json
{
  "passed": true,
  "overall_score": 8.5,
  "dimensions": {
    "code_quality": {
      "score": 9,
      "reason": "代码结构清晰，命名规范，有适当注释"
    },
    "test_coverage": {
      "score": 8,
      "reason": "测试覆盖主要场景，但边界情况不足"
    },
    "documentation": {
      "score": 8.5,
      "reason": "API文档完整，使用示例清晰"
    }
  },
  "summary": "整体质量良好，建议补充边界测试用例"
}
```

**字段说明**：

- **passed**：布尔值，整体是否通过
- **overall_score**：总分（0-10）
- **dimensions**：各维度评分
  - 每个维度包含 **score**（分数）和 **reason**（打分理由）
- **summary**：总体评价和建议

**这样的格式有什么好处？**

| 如果只问"代码好不好" | 如果限定输出格式 |
|---------------------|-----------------|
| 会给你散文式评论 | 会给你结构化数据 |
| 每次措辞不同 | 每次格式一致 |
| 无法比较 | 可以跨版本比较 |
| 无法进自动化流程 | 可以直接进 CI |

**核心理念**：**你收集的是评估数据，不是审稿意见。评估数据是给系统用的，会持续积累。**

### 两种评分器分工

| 评分器 | 职责 | 手段 | 上限 |
|--------|------|------|------|
| **确定性检查** | 抓底线——基础动作有无发生 | 纯代码规则 | 只能判断动作发生没有，判断不了结果好坏 |
| **评分细则检查** | 抓质量——做出来的东西像不像你要的 | LLM + 锁死输出格式 | 能判断质量，但有 LLM 的不确定性 |

**组合使用**：

```
先用确定性检查抓底线
         ↓
      通过了？
         ↓
再用评分细则检查抓质量
```

这套组合从一开始就设计成可以接进持续集成的。一旦进了流水线，不再依赖某个工程师记得手动去试，变成**持续运行的回归守门员**。

### 什么时候用哪种？

| 场景 | 用哪种检查 | 原因 |
|------|-----------|------|
| 判断文件是否创建 | 确定性检查 | 有明确的 True/False |
| 判断命令是否执行 | 确定性检查 | 有明确的 True/False |
| 判断代码质量 | 评分细则检查 | 质量是主观的，需要 Rubric |
| 判断是否符合团队规范 | 评分细则检查 | 规范可能有多种表达方式 |

## 💡 五条核心评估原则

前面展示了从"感觉更好"到"有证据"的转变：运行代理，记录发生的情况，并通过一组简单的检查进行评分。一旦建立了这样的循环，每次调整都更容易验证，任何倒退也都会变得清晰明了。

以下是评估的关键要点：

### 1. 只评估核心关键项

聚焦真正影响使用的行为，衡量真正重要的指标。

良好的评估能让回归结果清晰明了，让失败原因易于解释。

---

### 2. 先定义"完成标准"，再开发 Skill

先定义可校验的"完成标准"，再开发 Skill。

可以借助 `$skill-creator` 进行初始化，然后逐步收紧指令，直到成功结果变得明确无误。

---

### 3. 以实际运行行为为依据

评估以**实际运行行为**为依据，依托执行轨迹做客观判断。

不是看 Skill 写了什么，而是看 Skill 实际做了什么。

---

### 4. 硬性规则抓底线，评分细则抓质量

纯硬性规则覆盖不到的风格、规范类要求，借助智能体按评分细则质检。

确定性检查抓底线，评分细则检查抓质量。

---

### 5. 持续完善评估体系

线上、使用中出现的真实问题，都补充为测试用例，持续完善评估体系。

评估不是一次性的，而是持续积累的。

---

## 总结

今天我们学习了两类评分器：

1. **三步构建**：记录运行轨迹 → 确定性检查 → 评分细则检查
2. **确定性检查**：纯代码规则，抓底线，回答"动作发生了没有"
3. **评分细则检查**：LLM + 锁死格式，抓质量，回答"做出来的东西像不像你要的"
4. **核心理念**：收集的是评估数据，不是审稿意见

**关键**：确定性检查快速可靠但上限低，评分细则检查灵活但需要设计好 Rubric。两者组合使用，才能既抓底线又抓质量。
