
> 学习目标：理解 Skill 为什么能突破上下文窗口限制，比长 Prompt 更高效

---

## 引言

传统 Prompt 最大的问题是什么？

不管用不用，全部塞进上下文。

而 Skill 采用分层加载——用到才读，不用不载。

这一节我们拆解它的三级加载机制。

---

## 📊 传统 Prompt 的致命问题

### 一次性全量加载

传统长 Prompt 的做法：

```
System Prompt:
- 系统规则（1000 tokens）
- 用户历史对话（5000 tokens）
- 你的长 Prompt（50000 tokens）
- 其他 Prompt 的元数据（1000 tokens）
- 用户当前请求（100 tokens）

总计：57100 tokens
```

**问题**：
- 上下文窗口被一次性塞满
- 不管当前任务用不用这些信息，全部占用 Token
- Token 消耗极高，成本昂贵
- 上下文窗口有限，无法承载海量知识

---

## ✅ Skill 的渐进式披露机制

### 核心思想

**分层加载、按需披露、用到才读、不用不载**

---

### 三级加载架构

#### 第一层：元数据（常驻、轻量）

**内容**：所有 Skill 的 YAML frontmatter（name + description）

**Token 消耗**：约 100 tokens / Skill

**加载时机**：智能体启动时预加载，常驻上下文

**作用**：
- 让 AI 只看一句话简介，就能判断「当前任务要不要用这个技能」
- 极致节省 Token，不需要加载全文就能做路由判断

---

#### 第二层：SKILL.md 正文（按需加载）

**内容**：完整的任务流程、操作规范、使用说明

**Token 消耗**：约 5000 tokens

**加载时机**：当 AI 判断需要使用该技能时，才从文件系统读取完整内容载入上下文

**作用**：
- 指导 AI 完成标准作业
- 用到才加载，不用不占用 Token

---

#### 第三层：附属文件（精准加载）

**内容**：references/、scripts/、assets/ 等扩展文件

**Token 消耗**：按需计算

**加载时机**：执行到对应细分场景、需要对应信息时，才动态读取

**作用**：
- 补充 SKILL.md 无法覆盖的详细说明
- 进一步降低 Token 消耗
- 实现近似无限的上下文承载量

---

## 🎭 工作流程演示：用户触发 PDF Skill

### 阶段 1：初始状态（最小上下文）

上下文窗口包含：
- 核心系统提示（System Prompt）
- 每个已安装技能的 YAML 元数据（约 100 tokens / Skill）
- 用户的初始消息（`User：Fill out ...`）

**总消耗**：极少，大部分 Token 闲置可用

---

### 阶段 2：技能判定与触发

AI 通过元数据判断当前任务需要 PDF 技能。

**动作**：主动调用 Bash 工具，从文件系统读取 `pdf/SKILL.md` 完整内容载入上下文。

**Token 变化**：增加约 5000 tokens（SKILL.md 正文）

---

### 阶段 3：按需加载附加内容

如果任务需要填充表单，AI 继续读取附属文件 `FORMS.md`，补充细分规则进上下文。

**Token 变化**：增加约 2000 tokens（FORMS.md）

---

### 阶段 4：完整执行任务

上下文集齐所需全部规范，AI 按照 Skill 流程完成标准化任务。

---

## 📊 对比表：传统 Prompt vs Skill

| 维度 | 传统 Prompt | Skill 渐进式披露 |
|------|------------|----------------|
| **加载方式** | 一次性全量加载 | 分层按需加载 |
| **Token 消耗** | 全部占用，不管用不用 | 用到才占用，不用不载 |
| **上下文承载** | 受限于窗口大小 | 近似无限（TB级资料也能管理） |
| **成本控制** | 极高（每次全部消耗） | 极低（精准消耗） |
| **知识管理** | 无法沉淀，每次重写 | 文件化管理，持续迭代 |

---

## 💡 为什么能实现近似无限上下文？

### 核心机制：认知执行分离

**认知空间（上下文）**：AI 的思考空间，保留当前任务的微量信息

**执行空间（虚拟机）**：工具执行空间，通过 Bash/Read/Execute 动态调用文件系统

**突破原理**：
- 文件系统可以承载 TB 级资料
- AI 上下文只保留"当前任务的微量信息"
- 其他通过工具动态调用，不在上下文中

---

## 🎯 实战价值

### 1. 大型知识库 Skill

即使某个 Skill 包含 TB 级资料（代码库、数据库、文档），Agent 只需在上下文中保留当前任务的微量信息，其余通过工具动态调用。

### 2. 多 Skill 协作场景

假设你有 100 个 Skill：
- 传统 Prompt：100 个 Skill 的全部内容一次性塞进上下文 → 窗口爆满
- Skill 渐进式披露：只预加载 100 个元数据（100 × 100 = 10000 tokens）→ 窗口宽敞，按需加载

### 3. 持续迭代场景

每次改进 Skill，只需更新文件系统，不需要重写 Prompt，不需要重新训练模型。

---

## 🚨 实施要点

### SKILL.md 正文控制在 500 行以内

官方建议：SKILL.md 正文保持在 500 行以内以获得最佳性能。

如果超过此限制：
- 将详细内容拆分到单独的 reference 文件
- 在 SKILL.md 中用链接引用
- AI 按需读取，不默认加载

---

### 附属文件设计原则

**附属文件只在 SKILL.md 引用，不要嵌套超过一层**

```
✅ 正确：
SKILL.md → references/api.md
SKILL.md → references/examples.md

❌ 错误（嵌套两层）：
SKILL.md → advanced.md → details.md
```

原因：Agent 遇到嵌套引用时，可能会用 `head -100` 等命令预览内容，而不是读取整个文件，导致信息不完整。

---

## 🎯 本节核心观点

**渐进式披露是 Skill 碾压传统 Prompt 的核心设计原理：**
- 分层加载、按需披露、用到才读、不用不载
- 实现近似无限的上下文承载量
- 极致节省 Token，降低成本
- 突破大模型窗口限制
