
> 学习目标：建立 Agent 架构全景认知，掌握四大组件的正确使用场景

---

## 引言

你是不是也遇到过这种情况：

- 明明是临时任务，却写了个 Skill，结果用一次就扔，白白浪费工程化成本
- 明明是团队通用能力，却每次都写 Prompt，重复劳动，累死累活
- 想连接外部数据，不知道用 MCP 还是 Skill，搞了一堆混乱配置
- 把 Subagent 当 Skill 用，弄得对话乱七八糟，上下文混成一锅粥

**根本原因**：不知道 Prompt、Skill、Subagent、MCP 这四个东西到底有什么区别，什么时候该用哪个。

更扎心的是：

- 选错了工具，浪费时间降低效率
- 该沉淀的能力没沉淀，每次重新写
- 该隔离的任务没隔离，主对话被污染
- 该接外部资源没接，手动复制粘贴

今天这节导言，帮你一次性搞清楚：**这四个组件各司其职，互不重叠**。

搞清楚了，你就能：
- 选对工具，效率翻倍
- 沉淀能力，不再重复劳动
- 隔离任务，对话更清晰
- 连接资源，自动化程度更高

---

## 📊 四大组件定位总览

先看一张对比表：

| 组件 | 核心定位 | 典型场景 | 特点 |
|------|---------|---------|------|
| **Prompt** | 一次性临时指令 | 单次任务微调 | 临时性、不可复用 |
| **Skill** | 可复用专业能力库 | 通用能力沉淀 | 可复用、可迭代、可移植 |
| **Subagent** | 独立专项 AI 助手 | 独立专项任务 | 隔离性、专项性、独立上下文 |
| **MCP** | 外部数据通道 | 外部数据接入 | 连接性、资源访问 |

---

## 🔍 逐个拆解

### 1. Prompt = 一次性临时指令

**核心定位**：单次任务微调

**典型场景**：
- 临时改一下输出格式
- 单次任务加个特殊要求
- 试探性调整 AI 行为

**特点**：
- **临时性**：用完即弃，不复用
- **轻量性**：快速调整，无需工程化
- **灵活性**：随时改，随时试

**什么时候用**：
- 单次任务需要微调
- 快速试验新想法
- 不需要沉淀复用

---

### 2. Skill = 可复用专业能力库

**核心定位**：通用能力沉淀

**典型场景**：
- 团队通用的代码审查流程
- 标准化的报告生成模板
- 固化的故障排查 SOP

**特点**：
- **可复用**：写好一次，全团队用
- **可迭代**：踩坑持续固化，越来越稳
- **可移植**：换环境换 AI 都能用
- **工程化**：文件化管理，版本控制

**什么时候用**：
- 通用能力需要沉淀
- 团队多人需要复用
- 长期维护持续迭代

---

### 3. Subagent = 独立专项 AI 助手

**核心定位**：独立专项任务

**典型场景**：
- 复杂故障排查需要独立 Agent
- 多轮对话的专项任务
- 需要隔离上下文的任务

**特点**：
- **隔离性**：独立上下文，不干扰主对话
- **专项性**：专注单一任务，不混用
- **独立性**：独立决策、独立执行、独立输出

**什么时候用**：
- 任务复杂度高，需要独立上下文
- 任务需要专项处理，不适合混在主对话
- 需要隔离风险，避免主对话被污染

---

### 4. MCP = 外部数据通道

**核心定位**：外部数据接入

**典型场景**：
- 连接企业数据库
- 访问外部 API
- 接入公司内部系统

**特点**：
- **连接性**：连接外部资源，不包含业务逻辑
- **资源访问**：提供"能访问什么"，不提供"怎么做"
- **权限管理**：管理外部访问权限，不封装方法论

**什么时候用**：
- 需要连接外部数据源
- 需要访问外部系统
- 需要接入企业内部资源

---

## 🎯 组合口诀

**单次微调用 Prompt，通用能力沉淀用 Skills，独立专项任务用子智能体，外部数据接入靠 MCP。**

高级用法：**组合用**

- **Skill + MCP**：既懂方法论（Skill），又能访问外部资源（MCP）
- **Skill + Subagent**：复杂任务先用 Skill 封装流程，再交给 Subagent 独立执行
- **Prompt + Skill**：Skill 提供基础流程，Prompt 微调当前任务

---

## 📊 决策树：选哪个？

### 1. 这是一次性任务吗？
- **Yes** → 用 **Prompt**
- **No** → 继续判断

### 2. 需要长期复用沉淀吗？
- **Yes** → 用 **Skill**
- **No** → 继续判断

### 3. 需要独立上下文隔离吗？
- **Yes** → 用 **Subagent**
- **No** → 继续判断

### 4. 需要连接外部数据源吗？
- **Yes** → 用 **MCP**
- **No** → 回到 Prompt（单次轻量调整）

---

## 🎭 类比理解：软件开发

| 组件 | 类比 | 作用 |
|------|------|------|
| **Prompt** | 临时代码注释 | 单次调整 |
| **Skill** | 设计模式库 | 通用方法论 |
| **Subagent** | 独立微服务 | 独立专项服务 |
| **MCP** | 数据库驱动 | 连接外部资源 |

---

## 🔗 Skill 与 MCP 的互补关系（重点）

这是一个容易混淆的组合，需要特别说明：

### Skill 提供"方法论"（How）
- 教 Agent 如何审查合同
- 告诉 Agent 怎么分析数据
- 指导 Agent 如何排查故障

### MCP 提供"资源连接"（What）
- 连接企业数据库
- 访问合同管理系统
- 接入监控仪表板

### 组合使用
**Skill 指导方法 + MCP 提供资源 = 完整能力**

例子：合同审查任务
1. Skill 指导如何审查合同（方法论）
2. MCP 连接数据库提取合同（资源访问）
3. Skill 指导比对风险字段（方法论）
4. MCP 调用风险评估 API（资源访问）
5. Skill 指导生成审查报告（方法论）

---

## 🎯 本节核心观点

**四大组件各司其职，互不重叠：**
- Prompt：临时微调
- Skill：能力沉淀
- Subagent：专项隔离
- MCP：资源连接

**选错工具 = 浪费时间 + 降低效率**
