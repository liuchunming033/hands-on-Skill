
> 学习目标：掌握从草稿到迭代闭环的标准流程

---


## 引言

前面两节，我们理解了为什么需要评估，也知道了要分清 Skill 类型、找准评估方向。

但还有一个更实操的问题：**写 Skill 的流程怎么标准化？**

你是不是也这样写 Skill：写完 → 凭感觉试一下 → 发现问题 → 改 → 再试 → 觉得差不多了就收工。

这样写出来的 Skill，心里总是没底。改了几次都不知道是变好还是变坏，也不知道有没有把别处搞坏。

**根本原因**：没有标准流程，评估成了最后验收，而不是持续提升的前提。

**解决**：Anthropic 把写 Skill 变成七步闭环，让评估贯穿始终。

## 核心内容

### 七步生命周期全览

```
决定  → 草稿  → 测试  → 看结果  → 展示  → 改Skill  → 扩大
  ↓      ↓      ↓       ↓        ↓       ↓       ↓
定方向  写初稿  立即跑   看哪里偏   给人看   针对性改   加样本
```

这七步不是一次性跑完就结束，而是一个**循环**：改完 Skill 后回到第三步重新测试，确认旧问题被修掉。

### 第一步：决定 Skill 要做什么

**为什么这一步最重要？**

因为它决定后面所有评估的方向。如果连这个 Skill 要解决什么任务都说不清，后面就无法判断它有没有成功。

**要回答的问题**：

- 这个 Skill 要解决什么具体问题？
- 成功的标准是什么？
- 什么情况下算失败？

**示例**：

- ❌ 模糊："帮用户写 commit message"
- ✅ 具体："根据 git diff 生成符合团队规范的 commit message，包含 type(scope): description 格式，每次只描述一个改动"

**输出**：一个清晰的 Skill 定义，写在 SKILL.md 的 description 里。

### 第二步：写一个草稿

**特点**：SKILL.md 还没有到成品阶段，更像候选方案。

**不要追求完美**：先把流程、约束、需要 Agent 注意的细节写出来。

**草稿的三个要素**：

| 要素 | 内容 | 示例 |
|------|------|------|
| **流程** | Agent 应该怎么做 | 先分析 diff，再归类改动类型，最后生成 message |
| **约束** | 不能做什么 | 不要把多个改动混在一个 message 里 |
| **细节** | 容易出错的地方 | 如果 diff 超过 100 行，先总结再写 message |

**草稿状态**：可能漏东西，可能写得太宽，但有了它才能开始测试。

### 第三步：创建少量测试提示词，马上跑起来

**和普通写 prompt 习惯分开**：

| 普通做法 | Anthropic 做法 |
|---------|---------------|
| 写完凭感觉试一下 | 草稿一出现就立刻面对测试用例 |
| 试一个觉得"差不多"就收工 | 准备 3-5 个测试用例，覆盖不同场景 |

**测试用例设计**：

- **正常场景**：一个典型的 git diff
- **边界场景**：超长的 diff、多个文件改动
- **失败场景**：空 diff、不相关的改动

**关键**：测试用例本身就是资产，要保存下来。

#### evals.json 真实示例

来自 feishu-wiki-to-obsidian Skill 的评测文件：

**文件结构**：

```
feishu-wiki-to-obsidian/
├── SKILL.md
└── evals/
    └── evals.json    # 评测用例
```

**evals.json 内容**：

```json
[
  {
    "id": 1,
    "prompt": "帮我把这篇飞书 Wiki 导入到本地知识库：https://my.feishu.cn/wiki/P0Oewx6FpiU2NOkTUrJc7J3MnSb 放到 raw/01-articles 目录",
    "expected_output": "成功获取文档内容，提取标题，下载所有图片到 raw/assets/，保存 Markdown 到 raw/01-articles/，图片引用替换为本地路径",
    "files": []
  },
  {
    "id": 2,
    "prompt": "导入这个飞书文档到我的 Obsidian Vault：https://my.feishu.cn/wiki/OrYAw8Ex9iCEoakWvDncREh1nCb 图片下载到 raw/assets",
    "expected_output": "成功获取文档内容并保存，图片下载到 raw/assets 目录，Markdown 中图片链接替换为相对路径",
    "files": []
  },
  {
    "id": 3,
    "prompt": "https://my.feishu.cn/docx/AbCdEfGhIjKlMnOpQrStUvWxYz 把这个文档拉到本地的 raw/ 文件夹",
    "expected_output": "解析 docx URL，获取文档内容，保存到 raw/ 目录，处理图片",
    "files": []
  }
]
```

**评测用例设计要点**：

| 字段 | 用途 | 示例 |
|------|------|------|
| `id` | 用例编号，便于追踪 | 1, 2, 3... |
| `prompt` | 用户实际会说的触发语 | 包含具体的 URL 和目标目录 |
| `expected_output` | 预期行为描述 | 图片下载、路径替换等关键步骤 |
| `files` | 需要的测试文件 | 空数组表示不需要附件 |

### 第四步：看结果

**不止看通过还是失败**，还要看：

- 输出哪里对、哪里偏、哪里不稳定
- 这个测试用例本身写得好不好——有些时候错误不在 Skill，是你根本没有把什么叫好结果写清楚

**三个维度看结果**：

| 维度 | 问题 | 行动 |
|------|------|------|
| **正确性** | 输出是否符合预期？ | 记录哪些场景失败 |
| **稳定性** | 同样的输入，每次输出是否一致？ | 记录哪些场景不稳定 |
| **测试质量** | 测试用例本身是否清晰？ | 改进测试用例的预期 |

### 第五步：展示结果

**工具**：Anthropic 有 Eval Viewer，把输出指标、失败原因放到人面前。

**展示的内容**：

- 每个测试用例的输出
- 通过/失败状态
- 失败原因（如果有）
- 输出与预期的差异

### 第六步：根据反馈改 Skill

**进入迭代**：不再停留于"感觉这句话可以写得更顺"，而是追问：

- 哪个测试失败了？
- 失败原因是什么？
- 是流程漏了一步，还是约束写得太宽，还是示例不够？

**改完后再跑同一批测试**：确认旧问题被修掉。

**迭代的三种情况**：

| 情况 | 原因 | 行动 |
|------|------|------|
| 流程漏了一步 | Agent 跳过了某个环节 | 在 instructions 里加强这一步 |
| 约束太宽 | Agent 做了不该做的事 | 添加明确的否定约束 |
| 示例不够 | Agent 输出格式不稳定 | 加几个 input/output 示例 |

### 第七步：扩大测试集

少量测试只能证明你没有在几个例子上失败，不能证明这个 Skill 稳定。

**扩大的策略**：

- 从 3-5 个 → 20-50 个 → 持续积累
- 每次遇到失败案例,把它变成测试用例
- 让测试集随着时间越来越全面

---

### 评估集变大后，用并行评估提升效率

**问题**：测试集变大后，串行执行评估速度慢，不同测试用例之间上下文还会互相干扰。

**解决**：用多智能体并行运行评估。

Anthropic**新版 skill-creator 的并行能力**：

每个测试用例都会启动独立智能体，拥有干净的上下文环境，单独统计耗时与 Token 数据。

**好处**：

| 串行评估 | 并行评估 |
|---------|---------|
| 速度慢，耗时累加 | 速度快，多倍提速 |
| 上下文可能互相干扰 | 独立上下文，互不影响 |
| Token 数据可能混淆 | 单独统计，数据准确 |

**适用场景**：

- 测试集超过 20 个，需要提升效率
- 需要准确统计每个测试的耗时和 Token
- 需要保证测试环境干净，避免干扰

**核心价值**：既提升评估速度，也杜绝测试数据交叉影响。

### 把七步变成闭环

关键不是七步走完，而是**循环起来**：

```
改完 Skill → 回到第三步重跑测试 → 看结果 → 再改 → 再跑
```

**闭环的含义**：

- 每次改动都有测试验证
- 每次失败都能定位原因
- 每次修复都能确认有效

**评估不再是最后验收，变成 Skill 能持续提升的前提。**

## 总结

今天我们学习了七步生命周期：

1. **决定**：定义清晰的成功标准
2. **草稿**：先写出来，不追求完美
3. **测试**：立刻面对测试用例
4. **看结果**：正确性、稳定性、测试质量
5. **展示**：把结果放到人面前
6. **改 Skill**：针对性迭代
7. **扩大**：测试集持续增长

**核心理念**：把失败变成可复现的对象，Skill 才能进入迭代闭环。
