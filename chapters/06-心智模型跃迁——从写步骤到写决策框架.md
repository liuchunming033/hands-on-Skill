
> 学习目标：建立正确的 Skill 编写心智，从根本上理解 Skill 设计哲学

---

## 引言

很多人写 Skill 时，总觉得自己在写操作手册。

错了！

你写的是 AI 的决策框架。

这一节带你完成心智跃迁——从关注单次执行到关注系统能力，从写步骤清单到写约束边界。

---

## 💡 核心转变：从执行者到设计者

### 执行者的心智（错误）

**思维模式**：
- "我要教 AI 怎么做这件事"
- "我要把每一步都写清楚"
- "AI 应该按我的步骤执行"

**典型表现**：
- 写详细的步骤清单
- 每一步都给具体指令
- 追求"AI 忠实执行我的流程"

**问题**：
- AI 变成机械执行器，失去判断力
- 一遇到变化场景就失效
- 步骤越多，维护成本越高
- 无法适应 AI 自身的进步

---

### 设计者的心智（正确）

**思维模式**：
- "我要告诉 AI 不能做什么"
- "我要给 AI 组合能力，而不是固定步骤"
- "我要写决策框架，而不是操作清单"

**典型表现**：
- 写约束边界，告诉 AI 不能做什么
- 给组合能力，允许 AI 根据上下文选择路径
- 关注系统能力，而不是单次执行

**优势**：
- AI 保留判断力，能灵活应对变化
- 维护成本低，只需更新约束和知识
- 能适应 AI 自身的进步
- Skill 变成可迭代的系统资产

---

## 📊 四个核心框架

### 1. 触发边界（何时用）

**核心问题**：AI 如何判断是否应该使用这个 Skill？

**设计要点**：
- Description 写触发关键词和场景
- 避免"触发泛滥"（不该用的时候乱用）
- 避免"触发缺失"（该用的时候不用）

**示例**：
```yaml
description: "Generate weekly reports when user mentions '周报', 'weekly report',
'本周总结', or wants to summarize work progress."
```

---

### 2. 约束（不能做什么）

**核心问题**：AI 在执行过程中不能做什么？

**设计要点**：
- 写硬性约束（必须遵守的规则）
- 写软约束（建议遵守的规范）
- 避免"约束缺失"（AI 做出超出预期的操作）

**示例**：
```markdown
## Constraints

**Hard constraints (MUST follow)**:
- Do NOT delete any files without explicit user confirmation
- Do NOT modify production database directly
- Do NOT expose sensitive data in logs

**Soft constraints (SHOULD follow)**:
- Prefer active voice in output
- Use team conventions for naming
```

---

### 3. 结构（怎么组合）

**核心问题**：AI 如何组合使用这些能力？

**设计要点**：
- 给组合能力，而不是固定步骤
- 允许 AI 根据上下文选择路径
- 避免"细节堆砌"（把每一步都写死）

**示例**：
```markdown
## Workflow

**Core capabilities**:
1. Data collection (from Feishu Tasks, Calendar, or user input)
2. Data analysis (identify patterns, extract insights)
3. Report generation (create structured output)

**Combination principle**:
- Start with available data source
- Choose appropriate analysis method based on context
- Generate report following team template
```

---

### 4. 可测性（如何验证）

**核心问题**：如何验证 Skill 是否按预期工作？

**设计要点**：
- 写可检查的定义（什么叫成功）
- 设计测试用例（显式调用、隐式调用、负控样本）
- 避免"验证缺位"（不知道 Skill 是否有效）

**示例**：
```markdown
## Quality Guidelines

**Must satisfy**:
- Report contains required sections (本周工作、下周计划、心得体会)
- Report follows team formatting standards
- Report includes data from requested sources

**Test scenarios**:
- User explicitly requests report → should trigger
- User mentions "周报" → should trigger
- User asks unrelated question → should NOT trigger
```

## 🚨 四个常见陷阱

### 陷阱 1：细节堆砌

**表现**：把每一步都写清楚，AI 变成机械执行器

**后果**：
- AI 失去判断力
- 无法应对变化场景
- 维护成本极高

**解法**：给组合能力，允许 AI 灵活组合

---

### 陷阱 2：触发泛滥

**表现**：Description 写得太泛，不该用的时候乱触发

**后果**：
- AI 在不该用的场景强行使用 Skill
- 输出不符合预期
- 用户信任度下降

**解法**：写精确的触发关键词和场景

---

### 陷阱 3：约束缺失

**表现**：只写该做什么，不写不能做什么

**后果**：
- AI 做出超出预期的操作
- 可能删文件、改配置、泄露数据
- 高风险操作失控

**解法**：写硬性约束和软约束

---

### 陷阱 4：验证缺位

**表现**：写完 Skill 就不管了，不知道是否有效

**后果**：
- 凭感觉迭代，没有证据闭环
- 改了也不知道是优化还是回归
- Skill 变成未经检验的假设

**解法**：写可检查的定义，设计测试用例

## 🎯 四个心智跃迁

### 跃迁 1：写约束而非写指令

**Before（错误）**：
```markdown
## Step-by-step instructions

Step 1: Read the file
Step 2: Parse the data
Step 3: Extract the key information
Step 4: Generate the output
Step 5: Save the result
```

**After（正确）**：
```markdown
## Constraints

**Hard constraints**:
- Do NOT modify the original file
- Do NOT skip required fields

**Core capability**:
Extract and structure key information from files
```

---

### 跃迁 2：给组合能力而非固定步骤

**Before（错误）**：
```markdown
## Fixed workflow

Step 1 → Step 2 → Step 3 → Step 4
```

**After（正确）**：
```markdown
## Composable capabilities

**Capability A**: Data collection (from source X, Y, or Z)
**Capability B**: Data analysis (method 1, 2, or 3)
**Capability C**: Output generation (format A, B, or C)

AI chooses combination based on context.
```

---

### 跃迁 3：写决策框架而非操作清单

**Before（错误）**：
操作清单，AI 忠实执行固定流程。

**After（正确）**：
决策框架，AI 根据上下文判断应该怎么做。

---

### 跃迁 4：关注系统而非单次

**Before（错误）**：
关注单次执行是否正确。

**After（正确）**：
关注系统能力是否稳定、可测试、可迭代。

---

## 🎯 本节核心观点

**Skill 编写的根本转变：**
- 从执行者到设计者
- 从写步骤到写约束
- 从关注单次到关注系统
- 从操作清单到决策框架

**四个核心框架：**
- 触发边界（何时用）
- 约束（不能做什么）
- 结构（怎么组合）
- 可测性（如何验证）
