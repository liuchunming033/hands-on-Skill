
> 学习目标：掌握如何让 Agent 在信息不足时主动询问，而不是自己猜测

---

## 引言

Agent 太爱猜——一拿到模糊需求就急着生成，信息不够就自己补，补得非常自信。

比如你让 Agent："帮我搭建一个 React 项目"。

Agent 可能会猜：

- 要不要 TypeScript？
- 要不要路由？
- 用什么状态管理？
- 要不要配置 ESLint？

猜错了，你就要返工；猜对了，是运气。

**Inversion 模式**解决的就是这个问题。

---

## 📚 核心问题

**Agent 太爱猜，一拿到模糊需求就急着开工。**

常见表现：

- 用户说"帮我搭个项目"，Agent 直接开始生成，没问要什么技术栈
- 用户说"写个 API 文档"，Agent 直接写，没问是什么 API
- 用户说"分析这个数据"，Agent 直接分析，没问关注什么指标

---

## 💡 Inversion 的解决方案

**核心思想**：把流程反过来——Agent 先发问，用户先回答。
![inversion-pattern.png](images/inversion-pattern.png)

**三阶段设计**：

1. **Discovery（需求澄清）**：先确认用户到底要解决什么问题
2. **Constraints（约束确认）**：把平台、技术栈、限制条件、交付要求问完整
3. **Synthesis（开始生成）**：信息齐了之后才允许开始生成

**关键**：用门控指令明确规定——信息没齐不准开工。

## 🔧 Inversion SKILL.md 示例

**场景**：React 项目搭建 Skill

`````markdown
---
name: react-project-setup
description: Scaffold a new React project with customized configuration. Use when creating a new React app or setting up a React project.
---

# React 项目搭建

⚠️ **重要**：在开始生成之前，必须先完成 Discovery 和 Constraints 阶段。

## Phase 1: Discovery（需求澄清）

在开工前，先问清楚以下问题：

1. **项目目标**：
   - 是产品原型？还是生产级项目？
   - 是学习用？还是商业项目？

2. **技术选型**：
   - TypeScript 还是 JavaScript？
   - 需要路由吗？（React Router）
   - 状态管理选哪个？（Redux / Zustand / Context）

3. **工具链**：
   - 要配置 ESLint 吗？
   - 要配置 Prettier 吗？
   - 用什么测试框架？

## Phase 2: Constraints（约束确认）

确保以下信息完整：

````markdown
项目配置确认：
- [ ] 语言：TypeScript / JavaScript
- [ ] 路由：需要 / 不需要
- [ ] 状态管理：Redux / Zustand / Context / 不需要
- [ ] ESLint：需要 / 不需要
- [ ] Prettier：需要 / 不需要
- [ ] 测试：Jest / Vitest / 不需要

## Phase 3: Synthesis（开始生成）

✅ **门控条件**：只有 Phase 2 的所有选项都确认后，才能进入此阶段。

现在根据用户的选择，开始搭建项目...
````
`````

## 📌 经典案例

### 案例：数据库迁移脚本生成器

**场景**：团队需要生成数据库迁移脚本，支持多种数据库。

**问题**：用户只说"生成迁移脚本"，Agent 不知道：
- 从哪个数据库迁移？
- 迁移到哪个数据库？
- 迁移哪些表？
- 需要保留数据吗？

如果直接生成，会做出错误假设，导致脚本不可用。

**Inversion 方案**：

**SKILL.md 核心内容**：

````markdown
# 数据库迁移脚本生成器

⚠️ **重要**：必须先完成需求澄清，才能生成脚本。

## Phase 1: Discovery（需求澄清）

必须询问以下问题：

1. **源数据库**：
   - 当前使用什么数据库？MySQL / PostgreSQL / MongoDB / SQLite / 其他？
   - 版本是多少？

2. **目标数据库**：
   - 要迁移到什么数据库？MySQL / PostgreSQL / MongoDB / SQLite / 其他？

3. **迁移范围**：
   - 迁移所有表还是部分表？
   - 如果是部分表，请列出表名。

4. **数据处理**：
   - 需要保留原有数据吗？
   - 需要转换数据格式吗？（如日期格式、编码格式）

5. **约束条件**：
   - 迁移期间允许停机吗？
   - 需要支持回滚吗？

## Phase 2: Constraints（约束确认）

请确认以下信息：

```markdown
迁移配置确认：
- [ ] 源数据库：[类型] [版本]
- [ ] 目标数据库：[类型] [版本]
- [ ] 迁移表：[全部 / 指定表名]
- [ ] 数据保留：是 / 否
- [ ] 数据转换：[转换需求]
- [ ] 停机允许：是 / 否
- [ ] 回滚支持：是 / 否

## Phase 3: Synthesis（开始生成）

✅ **门控条件**：只有 Phase 2 的所有选项都确认后，才能生成脚本。

根据确认的信息，开始生成迁移脚本...
```
````

**案例解析**：

1. **主动询问**：信息不足时，Agent 先问清楚再动手
2. **门控机制**：明确必需信息，没齐不准开工
3. **避免猜测**：不假设用户的数据库类型、迁移范围等
4. **减少返工**：一份准确的脚本，而不是多次修改

**实际效果对比**：

| 方案           | 首次成功率       | 返工次数     | 用户满意度 |
| ------------ | ----------- | -------- | ----- |
| 直接生成         | 35%（大量假设错误） | 平均 4 次   | 低     |
| Inversion 模式 | 95%（信息完整）   | 平均 0.2 次 | 高     |

**典型对话流程**：

````
用户：生成迁移脚本

Agent：好的，我需要先了解一些信息：
1. 当前使用什么数据库？MySQL / PostgreSQL / MongoDB / SQLite？
2. 要迁移到什么数据库？
3. 迁移所有表还是部分表？

用户：MySQL 8.0 → PostgreSQL 15，迁移所有表

Agent：明白了。还有几个问题：
4. 需要保留原有数据吗？
5. 需要转换数据格式吗？

用户：保留数据，日期格式从 DATETIME 转为 TIMESTAMP

Agent：收到，让我确认一下：
- 源：MySQL 8.0
- 目标：PostgreSQL 15
- 范围：全部表
- 保留数据：是
- 日期转换：DATETIME → TIMESTAMP
确认无误后，我开始生成脚本。

用户：确认

Agent：[生成迁移脚本]
````

**团队反馈**：
"以前生成的迁移脚本总是各种问题，现在先问清楚再生成，一次成功。"  Inversion 模式的经典 Skill 案例

---

## ✅ 适用场景

- **项目搭建**：技术栈选择、配置项繁多
- **需求模糊的任务**：用户只说了目标，没说细节
- **多选项任务**：有多个合理选项，需要用户明确
- **容易返工的任务**：猜测错误代价高

---

## 🎯 本节核心观点

**Inversion 模式的三个关键**：

1. **主动询问**：信息不足时，先问再动手
2. **门控机制**：明确哪些信息是必需的，没齐不准开工
3. **三阶段设计**：Discovery → Constraints → Synthesis
