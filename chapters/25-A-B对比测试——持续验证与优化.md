## 引言

上一节我们学会了六类评估指标：通过率、Token消耗、执行时长、命令次数、重试率、错误分布。

有了指标还不够，**你还需要对比，才能知道 Skill 是变好还是变坏**。

你是不是也遇到过这种情况：

- 改了 SKILL.md，不知道是优化还是倒退
- 新模型上线，不知道 Skill 是否还能用
- 不确定加 Skill 是否真的比原生模型好

**根本原因**：只有单次评估，没有对比基准。

**解决**：用 A/B 对比测试，让变化一目了然。

---

## 📚 核心问题

**怎么对比 Skill 的不同版本、不同状态。**

常见问题：

- 改了 Skill，不知道变化方向
- 新模型上线，Skill 兼容性未知
- 无法向团队证明 Skill 的价值

---

## 💡 A/B 对比的解决方案

**核心思想**：同一个测试集，跑两遍，对比六类指标。

**三种对比模式**：

| 对比模式 | 用途 | 解决的问题 |
|---------|------|-----------|
| **版本对比** | Skill v1 vs v2 | 改完后是优化还是倒退 |
| **开关对比** | 有 Skill vs 无 Skill | Skill 是否真的比原生模型好 |
| **模型对比** | 模型 A + Skill vs 模型 B + Skill | 新模型是否兼容 Skill |

---

## 📊 实战案例 1：版本对比

**背景**：优化了一个 API 文档生成 Skill，想确认是否真的变好。

**对比方案**：

```bash
# Step 1：准备测试集（50 个 API）
test_cases = load_test_cases("api_docs_test_set.json")

# Step 2：跑 Skill v1
results_v1 = run_evaluation(
    skill="api-doc-generator@v1",
    test_cases=test_cases
)

# Step 3：跑 Skill v2
results_v2 = run_evaluation(
    skill="api-doc-generator@v2",
    test_cases=test_cases
)

# Step 4：对比指标
compare_results(results_v1, results_v2)
```

**对比结果**：

| 指标       | Skill v1 | Skill v2 | 变化     |
| -------- | -------- | -------- | ------ |
| 通过率      | 70%      | 84%      | ✅ +14% |
| Token 消耗 | 12500    | 8900     | ✅ -29% |
| 执行时长     | 45s      | 32s      | ✅ -29% |
| 命令次数     | 12       | 8        | ✅ -33% |
| 重试率      | 30%      | 6%       | ✅ -80% |
| 错误类型分布   | 类型错误12次，超时8次 | 类型错误3次，超时1次 | ✅ 全面减少 |

**结论**：Skill v2 全面优于 v1，通过率提升、成本下降、速度提升、稳定性提升，可以发布。

## 📌 实战案例 2：开关对比

**背景**：想证明 Skill 是否真的比原生模型好。

**对比方案**：

```bash
# 对比组 1：无 Skill（原生模型）
results_baseline = run_evaluation(
    skill=None,
    test_cases=test_cases
)

# 对比组 2：有 Skill
results_with_skill = run_evaluation(
    skill="code-reviewer",
    test_cases=test_cases
)

# 对比指标
compare_results(results_baseline, results_with_skill)
```

**对比结果**：

| 指标 | 无 Skill | 有 Skill | 变化 |
|------|----------|----------|------|
| 通过率 | 45% | 84% | ✅ +39% |
| Token 消耗 | 18500 | 8900 | ✅ -52% |
| 执行时长 | 78s | 32s | ✅ -59% |
| 重试率 | 60% | 6% | ✅ -90% |
| 错误类型分布 | 类型错误18次，超时12次 | 类型错误2次，超时1次 | ✅ 大幅减少 |

**结论**：Skill 让模型表现显著提升，通过率翻倍、成本减半、速度提升、稳定性大幅改善，证明 Skill 有真实价值。

## 🔧 实战案例 3：模型对比

**背景**：新模型上线，验证 Skill 是否还能用。

**对比方案**：

```bash
# 对比组 1：旧模型 + Skill
results_old_model = run_evaluation(
    model="claude-3.5",
    skill="pdf-form-filler",
    test_cases=test_cases
)

# 对比组 2：新模型 + Skill
results_new_model = run_evaluation(
    model="claude-4.0",
    skill="pdf-form-filler",
    test_cases=test_cases
)

# 对比指标
compare_results(results_old_model, results_new_model)
```

**对比结果**：

| 指标 | Claude 3.5 + Skill | Claude 4.0 + Skill | 变化 |
|------|-------------------|-------------------|------|
| 通过率 | 72% | 95% | ✅ +23% |
| Token 消耗 | 15000 | 9200 | ✅ -39% |
| 执行时长 | 52s | 28s | ✅ -46% |
| 重试率 | 25% | 2% | ✅ -92% |
| 错误类型分布 | 类型错误8次，超时5次 | 类型错误1次，超时0次 | ✅ 几乎消除 |

**结论**：新模型不仅兼容 Skill，表现还更好，可以放心升级。

## ✅ A/B 对比的适用场景

**什么时候需要对比？**

| 场景 | 对比模式 | 目的 |
|------|---------|------|
| **版本迭代** | Skill v1 vs v2 | 确认改对了方向 |
| **效果证明** | 有 Skill vs 无 Skill | 向团队证明价值 |
| **模型升级** | 模型 A vs 模型 B | 验证兼容性 |
| **持续监控** | 定期对比 | 发现性能退化 |

---

## 🎯 A/B 对比的关键

**三个核心原则**：

1. **同一个测试集**：保证对比公平，排除测试集差异
2. **多维度指标**：不只看通过率，还要看六类指标的变化
3. **持续监控**：定期运行对比，及时发现退化

**对比流程**：

```
准备测试集 → 跑 A 组 → 跑 B 组 → 对比六类指标 → 得出结论
```

---
