
> 学习目标：理解Skill 的长期维护与团队管理的核心概念和实践方法

---


## 引言

当一个团队有几百个 Skills 在跑时，问题来了：

- 怎么分发？
- 怎么管理？
- 怎么避免越做越乱？

Claude Code 核心开发者分享了 Skills 生态运营的三个关键机制。

**核心运营思路**：先把反馈回路搭起来，不是先追求完美系统。重点是先让 Skill 进入真实使用，再在反馈里慢慢长出来。

## 核心内容

### 第一个机制：分发策略

**两种做法**：

| 方式 | 适用场景 | 做法 | 优势 |
|------|---------|------|------|
| **放进代码库** | 小团队 | `.cod/skills` 目录，拉代码就能用 | 简单省事 |
| **做成插件进市场** | 大规模 | 谁需要谁装 | 系统轻，避免全包负担 |

**问题**：团队大时，塞进仓库的每个 Skill 都给系统加负担，大家明明只需一小部分却要背整包。

市场模式让团队自己决定装哪些，系统更轻。

**选择原则**：

- 小团队（几十人以内）→ 放进代码库
- 大规模（几百人以上）→ 做成插件进市场

### 第二个机制：市场管理

**不靠审核委员会，先放到试验区跑一跑再看结果**。

**筛选流程**：

1. 做了 Skill → 传到 GitHub 沙盒目录
2. 去 Slack 或论坛告诉大家可以试
3. 看有没有人真的用起来，有没有持续反馈
4. Skill 有人持续在用且确实有价值 → 作者提 PR 挪进正式市场

**关键判断**：Skill 做出来容易，做烂也容易。没有筛选机制，市场会堆满重复、模糊、没人维护的垃圾 Skill，好用的反而被埋掉。

**核心理念**：

- 不靠权威拍脑袋
- 先看有没有人持续在用
- 让真实使用来筛选价值

### 第三个机制：Skill 之间的依赖关系

**例子**：一个 Skill 负责上传文件，另一个 Skill 生成 CSV 并上传，后者依赖前者。

**轻量协作而非重依赖管理**：

- 不给一套很重的依赖管理系统（声明版本、装配关系、处理配置）
- 在 Skill 里写清楚"如果要上传文件，就调用哪个 Skill"
- 只要那个 Skill 已装好，Agent 自己去调用

**设计哲学**：简单，Skills 体系还在快速变化，先让东西能接起来、能跑起来，比上复杂机制更重要。

### 第四个机制：度量与数据驱动优化

**问题**：Skill 做出来后，不知道有没有人用、什么时候用、哪些该被调用的场景却没有被调用。

**解决方案**：PreToolUse Hook 记录使用情况——调用 Skill 前触发，记录时间、用户、Skill 名称、参数。

#### PreToolUse Hook 实现示例

**全局配置（settings.json）**：

```json
# ~/.claude/settings.json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Skill",
      "hooks": [{ "type": "command", "command": "~/.claude/hooks/log-skill.sh" }]
    }]
  }
}
```

**Hook 脚本（log-skill.sh）**：

```bash
# ~/.claude/hooks/log-skill.sh

#!/bin/bash
# stdin is the hook payload: { tool_name, tool_input: { skill, args }, session_id, ... }
# matcher already filtered to Skill, so no tool_name check needed
payload=$(cat)
skill=$(jq -r '.tool_input.skill' <<< "$payload")
args=$(jq -r '.tool_input.args // ""' <<< "$payload")

echo "$(date -u +%s)  $USER   $skill  $args" >> ~/.claude/skill-usage.tsv
```

#### 分析使用数据

有了记录能回答关键问题：

- 哪个 Skill 最受欢迎？
- 哪个 Skill 明明应该常用，但实际很少被触发？
- 还有哪些 Skill 基本没人碰，可能该下线了？

**使用场景**：

- 发现哪些 Skill 受欢迎
- 检测触发频率过低的情况
- 优化 Skill 库，淘汰低使用率的 Skill

**价值**：真正被记录的不只是使用数据，更重要的是团队有了判断依据。

凭感觉说 Skill 有用，现在能看到真实情况，决定是继续优化 description、补文档，还是删除。

### 核心运营思路总结

| 维度 | 做法 |
|------|------|
| **分发** | 按团队规模选方式（小→代码库，大→市场） |
| **筛选** | 不靠权威拍脑袋，先看有没有人持续在用 |
| **优化** | 不靠感觉，靠数据 |
| **迭代** | 先进入真实使用，再在反馈里慢慢长出来 |

**核心思想**：先把反馈回路搭起来，不是先追求把系统设计到完美。

Agent 内部很多 Skill 一开始也就几行——说明加一个踩坑点，然后边用边补。

**重点不是先规划出完美版本再发布，重点是先让它进入真实使用，再在反馈里慢慢长出来。**

### 避坑指南

**不要**：

- 一上来就想设计完美的依赖管理系统
- 让审核委员会决定哪些 Skill 能上线
- 凭感觉判断 Skill 有没有价值

**要**：

- 先让 Skill 进入真实使用
- 用数据驱动决策
- 让反馈来驱动迭代

## 总结

今天我们学习了 Skill 的长期维护与团队管理：

1. **分发策略**：小团队放代码库，大规模用市场模式
2. **市场管理**：试验区筛选而非审核委员会
3. **依赖关系**：轻量协作而非重依赖管理
4. **度量优化**：pre-use hook 记录使用情况，数据驱动决策

**核心思想**：先让 Skill 进入真实使用，再在反馈里慢慢长出来。
