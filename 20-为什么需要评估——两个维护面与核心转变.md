
> 学习目标：理解评估对Skill的重要性，没有评估只是假设

---


## 引言

你是不是也遇到过这种情况：改了几次 Skill，每次跑完看起来没问题，但心里总觉得不踏实？

更扎心的是这几个问题：

- 改完了，怎么知道真的变好了？
- 怎么知道不是只在这个例子上好用，其他场景搞坏了？
- 过段时间发现某个场景突然不好使了，但不知道是哪次改坏的？

**根本原因**：Skill 有两个独立维护面，任何一个面都可能失控。如果你只盯着其中一个面，另一个面的腐烂根本看不见。

**解决**：用评估把"我相信"变成"我可以证明"。

## 核心内容

### Skill 一旦上线，改不动了

Skill 一旦接入真实工作流，会触发工具、生成文件、留下副作用。改任何一个地方，整个行为边界都可能一起飘。

**凭感觉改 Skill 的典型场景**：

- 改了一版 Skill，跑了一下，看起来没出事，就继续改
- 靠的是体感，不是证据
- 过段时间发现：某个场景突然不好使了，但不知道是哪次改坏的

**为什么这么难追踪？**

因为 Skill 有两个独立的维护面，可以独立失控。

### 两个维护面：独立失控的风险

Skill 不是铁板一块，它有两个独立的维护面：

| 维护面 | 内容 | 失控表现 |
|-------|------|---------|
| **执行体** | instructions、工具链、步骤 | 改了 instructions，触发边界没变，但行为变了 |
| **触发边界** | name 和 description | 改了 description，执行体没变，但触发范围飘了 |

**执行体失控**：改了 SKILL.md 里的步骤说明，Skill 还是会在同样的场景触发，但做出来的事情已经变了。比如把"先分析再写代码"改成"直接写代码"，结果质量下降，但触发没变。

**触发边界失控**：改了 description，执行体没动，但 Skill 触发的范围变了。比如把"处理 PDF 表单"改成"处理 PDF"，突然所有 PDF 任务都会触发这个 Skill，而你的 Skill 根本不是为解析设计的。

**问题**：如果只盯着执行体，触发边界的腐烂根本看不见。反过来也一样。

### 三个评测场景：分别回答什么问题？

评估不是只有一种形态，OpenAI 把它分成三类场景：

| 场景 | 回答的问题 | 典型用法 |
|------|-----------|---------|
| **基准测试（Benchmark）** | 这个 Skill 有没必要存在？ | 比较 with skill 和 without skill，证明增量 |
| **回归测试（Regression Test）** | 我改了 Skill 以后，有没有把原来能做的事弄坏？ | 每次改 SKILL.md 都跑一遍，守住旧能力 |
| **版本对比（Version Comparison）** | 这一次改动到底有没有把 Skill 改好？ | A/B Testing，两个版本并排比较 |

**三类场景的顺序**：

```
基准测试 → 回归测试 → 版本对比
   ↓           ↓           ↓
证明有用   守住旧能力   决定往哪改
```

**基准测试示例**：

- 问题："我写了个 Skill 帮 Agent 写 commit message，但它本身就会写，我这个 Skill 有什么用？"
- 方法：准备 50 个 git diff，让 Agent 在有 Skill 和没 Skill 两种情况下写 commit message
- 结果：有 Skill 的版本明显更符合团队规范，证明 Skill 有增量

**回归测试示例**：

- 问题："我改了 commit message Skill 的格式说明，但突然发现另一个项目的 commit message 乱了"
- 方法：每次改 Skill 都跑一遍测试集
- 结果：发现改坏后立刻回滚，守住旧能力

### 核心转变：从信仰到证据

Anthropic 把这个转变总结成一张对比表：

| 没有评估时 | 有了评估以后 |
|-----------|------------|
| 写 Skill 像写信条——你相信这段流程会让 Agent 更好 | 我可以证明它在这些场景下有效 |
| 你相信这些规则会被遵守 | 我可以证明它比不用 Skill 更好 |
| 但这些都没有被验证 | 我可以证明修改没有破坏旧能力 |

**成熟的 Skill**：可以被验证、被比较、被复盘、被淘汰，也可以被持续迭代。

**不成熟的 Skill**：只有表面上的专业感，但没有证据支持。

### Eval 的第一层价值

目标不是给 Skill 打漂亮分数，而是**把失败变成可复现的对象**。

**为什么这很重要？**

- 只要失败可以复现，就可以定位
- 只要可以定位，就可以修复
- 只要修复后还能再测，Skill 才进入迭代闭环

**Anthropic 案例**：

PDF Skill 在处理非填充式表单时会出问题。如果没有 Eval，大概率只能靠偶然发现。有了 Eval，失败模式就被隔离出来——测试用例告诉你在这个场景里失败了。然后修复定位逻辑，修完后再跑 Eval，确认这个失败模式被解决。

## 总结

今天我们学习了为什么需要评估：

1. **两个维护面**：执行体和触发边界，可以独立失控
2. **三个评测场景**：基准测试证明有用、回归测试守住旧能力、版本对比决定往哪改
3. **核心转变**：从"我相信"变成"我可以证明"
4. **Eval 的价值**：把失败变成可复现的对象，进入迭代闭环

没有 Eval 的 Skill，只是一份还没被检验过的假设。
